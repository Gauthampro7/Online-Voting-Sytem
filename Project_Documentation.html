Online National Polling System

A comprehensive, highly secure, and aesthetically premium Java-based web application designed to facilitate democratic online voting, candidate nomination, and real-time result tracking.

Project File Structure

A well-organized project structure is the backbone of any Spring Boot application. We follow the Layered Architecture pattern to separate concerns and ensure maintainability.

📂

src/main/java (The Source Code)

This directory contains the logic of the application, organized into meaningful packages:

  • com.dsu.onlinevoting.config: Contains SecurityConfig.java, where we define access rules and password encryption beans.
  • com.dsu.onlinevoting.controller: The "Traffic Police" of the app. These classes (like VoterController.java) handle incoming web requests and return HTML pages.
  • com.dsu.onlinevoting.model: Our Entities. These classes define the structure of our database tables using JPA annotations.
  • com.dsu.onlinevoting.repository: Interfaces that talk to the database. They extend JpaRepository to provide CRUD operations without writing SQL.
  • com.dsu.onlinevoting.service: The "Brain" of the app. This layer contains business logic, such as calculating results or verifying vote eligibility.
📄

src/main/resources (The Assets)

  • templates/: Contains all Thymeleaf HTML files. These are transformed into dynamic web pages at runtime.
  • static/: Holds our custom styles.css and any images/icons used in the design.
  • application.properties: The central configuration file for database URLs, ports, and Hibernate settings.

Maven & Dependency Management

Modern Java development relies on Maven to manage project lifecycles. Instead of manually downloading JAR files, we define our needs in the pom.xml (Project Object Model) file.

Why use Maven?

Maven provides several critical benefits for a college project of this scale:

  1. Dependency Resolution: Automatically downloads all libraries (Spring, Hibernate, MySQL Driver) from the central repository.
  2. Build Automation: Compiles, tests, and packages the code into a runnable .jar file with a single command.
  3. Reproducibility: Ensures that if another developer (or your professor) runs the project, they get exactly the same environment.

Key Dependencies in pom.xml

The Request Lifecycle

When a user clicks "Vote", the request first hits the Embedded Tomcat Server. Spring Security intercepts it to check if the user is logged in. If allowed, the DispatcherServlet (the front controller) sends the request to the appropriate Controller. The Controller talks to the Service, which uses the Repository to save the vote in MySQL. Finally, a success message is passed back to the Thymeleaf template for rendering.

User Roles & Workflows

To simulate a real-world national polling environment, we implemented 4 distinct actors with unique permissions and responsibilities.

1

Voter (General Public)

The primary user. They can register, apply for a digital Voter ID by submitting their address and constituency, and once approved by an officer, they can cast a single vote. They also have access to the discussion forums to engage with other citizens.

2

Candidate

Users who wish to run for office. They must submit a candidacy application including their political party and manifesto. Once an Admin approves their candidacy, their name appears on the ballot for voters in their constituency.

3

Field Officer

The verification layer. Field officers are responsible for reviewing Voter ID applications. They check the provided documents and click "Approve" to grant the user voting rights, ensuring no duplicate or fake identities enter the system.

4

Administrator

The Election Commission. Admins have the highest authority. They approve candidate applications, monitor the live tally of votes, and oversee the overall health of the discussion forums. They are the only ones who can see the real-time winner of the election.

Database & JDBC Connectivity

To persist user data, candidate profiles, and immutable voting records, we utilized a MySQL relational database. Instead of writing raw, boilerplate JDBC queries (like DriverManager.getConnection() and PreparedStatement), we leveraged Spring Data JPA and Hibernate. This acts as an Object-Relational Mapping (ORM) layer.

1. application.properties Configuration

This is the core configuration file that establishes the JDBC connection pool (HikariCP) between our Java application and the MySQL server.

# Database Configuration
spring.datasource.url=jdbc:mysql://localhost:3306/online_voting?useSSL=false&serverTimezone=UTC
spring.datasource.username=root
spring.datasource.password=
spring.jpa.hibernate.ddl-auto=update
spring.jpa.properties.hibernate.dialect=org.hibernate.dialect.MySQLDialect

Explanation: ddl-auto=update is a powerful feature that automatically analyzes our Java classes and generates the SQL `CREATE TABLE` commands in MySQL for us. We use HikariCP (the default connection pool) to maintain a set of open connections to MySQL, which significantly improves performance by avoiding the overhead of creating a new connection for every single query.

The MySQLDialect ensures that Hibernate generates SQL syntax that is specific and optimized for the MySQL database engine, handling things like pagination and auto-incrementing IDs correctly.

2. Entity Modeling (The 'M' in MVC)

We mapped our database tables directly to Java Objects (POJOs) using JPA annotations. For example, the Vote table links a voter to a candidate.

@Entity
public class Vote {
    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long id;

    @ManyToOne // Foreign Key to the Users table
    @JoinColumn(name = "voter_id", nullable = false)
    private User voter;

    @ManyToOne // Foreign Key to the Users table
    @JoinColumn(name = "candidate_id", nullable = false)
    private User candidate;
    
    private LocalDateTime timestamp;
}

Backend Logic (Spring Boot)

The backend is divided into Repositories (Data Access) and Services (Business Logic).

A

Repository Interface

By extending JpaRepository, we instantly inherit methods like save(), findAll(), and findById() without writing SQL.

public interface VoteRepository extends JpaRepository<Vote, Long> {
    // Spring dynamically generates a "SELECT * FROM vote WHERE voter_id = ?" query!
    Optional<Vote> findByVoter(User voter);
    long countByCandidate(User candidate);
}

We use Optional<T> to handle cases where a record might not exist (preventing the dreaded NullPointerException). This encourages safer coding practices throughout the backend.

B

Service Layer

This is where the actual logic resides. We use @Transactional to ensure that if something fails during a multi-step database process, the entire transaction rolls back to prevent corrupted data.

@Service
public class VotingService {
    @Transactional
    public boolean castVote(User voter, User candidate) {
        if (!hasVoted(voter)) {
            Vote vote = new Vote(voter, candidate);
            voteRepository.save(vote); // Inserts into MySQL via JDBC
            return true;
        }
        return false;
    }
}

Frontend Design (Thymeleaf & Glassmorphism)

For the user interface, we avoided heavy Javascript frameworks to keep the project completely Java-centric. We used Thymeleaf, a server-side Java template engine that allows us to inject Java variables directly into HTML.

Thymeleaf Integration

We pass data from the Controller to the View using the Model object. Below is how we loop through the approved candidates and display them to the voter.

Security & Authorization

Elections require strict security. We implemented Spring Security to handle authentication, password hashing (BCrypt), and authorization.

Role Hierarchy

We defined 4 distinct roles using a Java Enum: ADMIN, FIELD_OFFICER, CANDIDATE, and VOTER. Spring Security is configured to intercept HTTP requests and block unauthorized access based on these roles.

@Bean
public SecurityFilterChain securityFilterChain(HttpSecurity http) throws Exception {
    http.authorizeHttpRequests((requests) -> requests
            // Public routes
            .requestMatchers("/", "/register", "/css/**").permitAll()
            
            // Protected routes based on Role
            .requestMatchers("/admin/**").hasRole("ADMIN")
            .requestMatchers("/field-officer/**").hasRole("FIELD_OFFICER")
            .requestMatchers("/voter/**").hasAnyRole("VOTER", "ADMIN")
            
            // Block everything else unless logged in
            .anyRequest().authenticated()
        )
        .formLogin((form) -> form.loginPage("/login").defaultSuccessUrl("/dashboard", true));

    return http.build();
}

This guarantees that a normal Voter cannot manually navigate to /admin/results or alter candidate approvals.

Security Highlights

  • BCrypt Hashing: Passwords are never stored in plain text. BCrypt uses an adaptive hashing algorithm with a salt, making it extremely resistant to rainbow table attacks.
  • CSRF Protection: Spring Security automatically adds a hidden CSRF token to every form, preventing "Cross-Site Request Forgery" attacks.
  • Session Management: Secure session handling ensures that users are automatically logged out after inactivity.

Future Scope & Enhancements

While the current system is fully functional for a graduation project, it can be expanded into a production-ready national platform with the following enhancements:

  • Blockchain Integration: Implementing a private blockchain (like Hyperledger) to store votes as immutable blocks, making the election completely audit-proof and tamper-resistant.
  • Biometric Verification: Integrating Aadhaar or Fingerprint APIs to verify the voter's identity in real-time during the login process.
  • Live Notification System: Using WebSockets to push real-time voting trends and results to the dashboard without requiring a page refresh.
  • AI Manifesto Analysis: Using Natural Language Processing (NLP) to summarize candidate manifestos and highlight key promises for voters.
