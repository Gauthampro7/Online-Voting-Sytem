Final Year Project

Online National Polling System

A fully functional, role-based electronic voting system built as a Java command-line application with a MySQL database. No web browser or framework needed — just run and interact in the terminal.

Project at a Glance

  • Language: Java 17
  • Database: MySQL / MariaDB (via JDBC)
  • Interface: Terminal / Command Line (CLI)
  • Build Tool: Maven (for dependency management only)
  • Package: com.dsu.onlinevoting

Technology Stack

The design priority was simplicity. Every technology choice was made to keep the code easy to read and explain.

Java (Language)

Java is the core programming language. We use Java 17, which supports modern features like text blocks ("""...""") for writing multi-line SQL cleanly, and the arrow-style switch statement for menus.

JDBC (Database Connection)

JDBC (Java Database Connectivity) is Java's built-in way to talk to databases. Instead of a framework, we use raw JDBC: we write SQL queries as strings and send them directly. This is more work than an ORM, but you can see exactly what SQL is being run at all times — great for learning.

MySQL / MariaDB (Database)

The data is stored in a MySQL-compatible database (we use the bundled portable MariaDB). MySQL is relational, meaning data is organised into tables with rows and columns — perfect for structured voting data.

Packages & Models

Even though this is a CLI application, it follows standard Java conventions for packaging and data modeling.

Package: com.dsu.onlinevoting

The application is organized under the com.dsu.onlinevoting package. This identifies the code as part of the "Dayananda Sagar University" (DSU) Online Voting project and ensures there are no name conflicts with other Java libraries.

Mental Models (Data Structures)

In this simplified architecture, we don't use separate "Model" classes (like User.java). Instead, the Database Tables serve as our Models. Each table represents a core concept:

  • User Model: Stores login credentials and role assignment.
  • Voter Profile Model: Stores citizen details and verification status.
  • Candidate Profile Model: Stores political party info and manifestos.
  • Vote Model: Represents the link between a voter and their chosen candidate.
  • Forum Post Model: Represents a message in the community discussion board.

Database Design

The application uses 5 tables. They are created automatically on first run — no manual SQL setup needed.

users

The central table. Every person in the system is a user — whether voter, candidate, officer, or admin.

users (
    id            INT  PRIMARY KEY AUTO_INCREMENT,
    username      VARCHAR(100) UNIQUE NOT NULL,  -- login name
    password_hash VARCHAR(64)  NOT NULL,         -- SHA-256 hash, never plain text
    name          VARCHAR(200) NOT NULL,
    role          ENUM('VOTER','CANDIDATE','FIELD_OFFICER','ADMIN')
)

voter_profiles

Stores a voter's Voter ID application. Linked to users via user_id. One voter can only have one profile (UNIQUE constraint).

voter_profiles (
    id           INT PRIMARY KEY AUTO_INCREMENT,
    user_id      INT UNIQUE,    -- Foreign key to users.id
    constituency VARCHAR(200),
    address      TEXT,
    status       ENUM('PENDING','APPROVED','REJECTED') DEFAULT 'PENDING'
)

candidate_profiles

Stores a candidate's election application. Same structure as voter_profiles but with party and manifesto fields.

candidate_profiles (
    id           INT PRIMARY KEY AUTO_INCREMENT,
    user_id      INT UNIQUE,
    party        VARCHAR(200),
    constituency VARCHAR(200),
    manifesto    TEXT,
    status       ENUM('PENDING','APPROVED','REJECTED') DEFAULT 'PENDING'
)

votes

Stores each cast vote. voter_id is UNIQUE — this is how we enforce the one-person-one-vote rule at the database level.

votes (
    id           INT PRIMARY KEY AUTO_INCREMENT,
    voter_id     INT UNIQUE,   -- UNIQUE prevents double voting
    candidate_id INT,
    voted_at     TIMESTAMP DEFAULT CURRENT_TIMESTAMP
)

forum_posts

Stores discussion messages. No UNIQUE constraint — users can post multiple times.

forum_posts (
    id        INT PRIMARY KEY AUTO_INCREMENT,
    user_id   INT,
    content   TEXT,
    posted_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
)

User Roles

There are 4 roles in the system. Each role sees a different menu when they log in.

🗳️

VOTER

The general public. Must apply for a Voter ID (approved by an officer) before they can vote. Can vote once, post on forum.

🎙️

CANDIDATE

Citizens running for office. Submit party name, constituency, and manifesto. Must be approved by Admin to appear on ballot.

🔍

FIELD_OFFICER

Verifies voter identity. Can only see and approve/reject voter ID applications. Cannot touch candidates or results.

👑

ADMIN

The Election Commission. Approves both voters and candidates. Can view the live vote tally ranked by winner.

Code Structure Explained

Everything is inside OnlineVotingApp.java. Here is what each section does:

1. Imports (top of file)

import java.security.MessageDigest; // For SHA-256 hashing
import java.sql.*;                  // JDBC: Connection, Statement, ResultSet, etc.
import java.util.Scanner;           // Reads user input from the terminal

2. DB Config Constants

private static final String DB_URL_BOOT = "jdbc:mysql://localhost:3306/...";
private static final String DB_URL      = "jdbc:mysql://localhost:3306/online_voting...";

Two URLs are used. DB_URL_BOOT connects without specifying a database (needed to run CREATE DATABASE). DB_URL connects with the database already selected for all normal operations.

3. main() — Program Entry Point

Runs at startup. Creates the DB if needed, then calls setupDatabase() to create tables, then enters the main login/register loop.

4. login() and register()

login() queries the users table with a hashed password. If found, it routes to the correct role menu. register() inserts a new user row with a hashed password. Both use PreparedStatement to prevent SQL Injection.

5. Role Menus (adminMenu, voterMenu, etc.)

Each is a while(true) loop that keeps showing the menu until the user picks "Logout" (which calls return to exit the loop).

6. Feature Methods

These are the actual operations: castVote(), approveReject(), viewResults(), etc. Each one builds a SQL query, executes it, and prints the result.

7. setupDatabase()

Uses CREATE TABLE IF NOT EXISTS for every table — safe to run every startup without losing any data.

8. sha256()

A utility method that converts a password string into a 64-character hex hash. Called whenever a password is saved or compared — the raw password is never stored.

Project Architecture

While this project is a single-file CLI application, it is designed with a "MVC-lite" (Model-View-Controller) architecture in mind.

The MVC-Lite Pattern

  • Model (The Database): All data logic is handled by the MySQL tables. The database ensures data integrity through UNIQUE constraints and Foreign Keys.
  • View (The Terminal): The Console acts as our interface. Every System.out.println() call is part of our "View" layer, presenting information to the user.
  • Controller (The Java Methods): Methods like castVote() or login() act as Controllers. They take input from the View, process it using the Model, and update the View again.

Why the Single-File Approach?

In a standard professional environment, we would use frameworks like Spring Boot to split these into 50+ files. However, for a learning project, the single-file approach is superior because:

  • Tracing Logic: You can follow a user's journey from login to voting by simply scrolling down, without jumping between dozens of tabs.
  • Zero Dependencies: No complex annotations (@Service, @Entity) are needed. It's just pure, raw Java.
  • Portability: You can copy just one .java file and it contains 100% of the application logic.

Security

Password Hashing (SHA-256)

Passwords are never stored in plain text. When a user registers, their password is run through SHA-256 (a one-way mathematical function). The resulting 64-character string is what gets saved to the database. When they log in, the typed password is hashed again and compared. This means even if someone reads the database, they cannot see real passwords.

SQL Injection Prevention

All user inputs are passed through PreparedStatement with ? placeholders. This prevents SQL Injection — a common attack where malicious input like ' OR 1=1 -- is typed to manipulate queries. With PreparedStatement, the input is treated as data, not SQL code.

One-Vote Enforcement

The votes table has a UNIQUE constraint on voter_id. Even if the application code had a bug, the database itself would reject a second vote from the same person with a duplicate key error.

Role-Based Access

After login, the user's role field from the database determines which menu they see. A VOTER cannot reach admin functions — those methods are simply never called from the voter menu.

How to Run

Step 1 — Start the Database

Open a terminal and run this command. Keep this terminal open while using the app.

cd "C:\Users\gauth\Online Voting Sytem\mariadb\mariadb-10.6.14-winx64"
.\bin\mysqld.exe --defaults-file=data\my.ini

Step 2 — Run the App (Eclipse)

  1. Right-click the project → Maven → Update Project (first time only)
  2. Right-click OnlineVotingApp.java → Run As → Java Application
  3. Type in the Console tab at the bottom of Eclipse

Step 3 — Run the App (Terminal / run.bat)

Double-click run.bat from the project folder, or compile and run manually:

javac -cp "mysql-connector.jar" -d target\classes src\main\java\...\OnlineVotingApp.java
java -cp "mysql-connector.jar;target\classes" com.dsu.onlinevoting.OnlineVotingApp

First Launch Behaviour

On the very first run, the app will automatically create the online_voting database and all 5 tables. It will also create the default admin account. No manual SQL needed.

Future Scope

  • Web Interface: Rebuild the CLI as a Spring Boot + Thymeleaf web app for browser access.
  • OTP Verification: Send a one-time password to the voter's phone/email to verify identity before voting.
  • Constituency Filtering: Only show candidates from the voter's own constituency when casting a vote.
  • Blockchain Audit Log: Store each vote as an immutable blockchain record for tamper-proof verification.
  • BCrypt Passwords: Upgrade from SHA-256 to BCrypt for stronger password protection (includes a salt).
  • Connection Pooling: Replace the single shared Connection with HikariCP for multi-user concurrency.
